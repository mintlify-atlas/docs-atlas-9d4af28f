## Overview

Not all switches support Layer 3 routing. SECURE-BRANCH-lac automatically detects whether each switch has `ip routing` enabled and adapts the generated configuration accordingly.

<Warning>
Attempting to configure Layer 3 features (like SVIs with IP addresses or DHCP pools) on a Layer 2-only switch will cause configuration errors.
</Warning>

## Detection Mechanism

### How Detection Works

The detection is performed by the `detect_l3()` function in `core/connector.py`:

```python
def detect_l3(nc) -> bool:
    """
    Detects if the switch has ip routing active (Layer 3) or not (Layer 2).

    Strategy: executes 'show ip route connected' and searches for lines
    that begin with 'C' (directly connected route). If found, the switch routes.

    Parameters
    ----------
    nc : Active Netmiko connection (in enable mode).

    Returns
    -------
    bool : True = Layer 3 (ip routing active), False = Layer 2.
    """
    try:
        out = nc.send_command("show ip route connected", delay_factor=2)
        return bool(re.search(r'^C\s+', out, re.MULTILINE))
    except Exception:
        return False   # If it fails, assume L2 for safety
```

## Detection in Practice

The detection happens automatically during the connection phase:

```python
def run_on_switch(sw: dict, config_params: dict, do_clean: bool = False, log_fn=None) -> bool:
    """
    Connects to a switch and applies complete configuration.
    """
    try:
        nc       = ConnectHandler(**make_device_params(sw))
        nc.enable()
        hostname = nc.find_prompt().replace("#", "").strip()
        is_l3    = detect_l3(nc)  # ← DETECTION HAPPENS HERE
        layer    = "Layer 3" if is_l3 else "Layer 2"
        if log_fn:
            log_fn(f"  ✔ Connected: '{hostname}'  |  {layer}")

        # ... rest of configuration deployment
        cmds = build_commands(is_l3=is_l3, log_fn=log_fn, **config_params)
```

## Common Scenarios

<Tabs>
  <Tab title="Branch Switch (L3)">
    **Use Case:** Full-featured branch switch with routing

    **Capabilities:**
    - Inter-VLAN routing
    - DHCP server for local VLANs
    - Static routes and OSPF
    - SVI IP addresses as gateways

    **Configuration Generated:**
    - `ip routing`
    - DHCP pools
    - SVIs with IPs
    - Routes
    - VLANs + ports
    - QoS
    - Security
  </Tab>

  <Tab title="Access Switch (L2)">
    **Use Case:** Simple access layer switch

    **Capabilities:**
    - VLAN segmentation
    - Port assignment (access/trunk)
    - QoS policies
    - Security features

    **Configuration Generated:**
    - VLANs + ports
    - QoS
    - Security
    - **NO** `ip routing`, DHCP pools, or SVI IPs

    **Note:** DHCP and routing handled by upstream router (R1)
  </Tab>

  <Tab title="Mixed Environment">
    **Use Case:** Some branches have L3 switches, others L2

    **Behavior:**
    - Each switch is detected independently
    - L3 switches get full config
    - L2 switches get limited config
    - Same configuration data (VLANs, QoS) is used for all

    **Advantage:** Single configuration can deploy to heterogeneous infrastructure
  </Tab>
</Tabs>

## Troubleshooting Detection

<AccordionGroup>
  <Accordion title="Detection Returns False on L3 Switch">
    **Possible Causes:**
    - `ip routing` not enabled in running-config
    - No SVIs with IP addresses configured yet
    - SSH timeout or permission issue

    **Solution:**
    - Manually verify: `show ip route connected`
    - Enable routing: `ip routing` in global config
    - Check user has privilege level 15
  </Accordion>

  <Accordion title="Commands Fail on L2 Switch">
    **Symptoms:**
    ```
    % Invalid input detected at '^' marker.
    ```

    **Cause:** App attempted to send L3 commands (usually a detection bug)

    **Solution:**
    - Check logs for detection result
    - Verify `show ip route connected` output manually
    - Report bug if detection is incorrect
  </Accordion>

  <Accordion title="Detection Times Out">
    **Symptoms:**
    - Connection succeeds but detection hangs

    **Cause:**
    - Switch is very slow or overloaded
    - Network latency

    **Solution:**
    - Increase `delay_factor` in `detect_l3()` function
    - Check switch CPU usage: `show processes cpu`
  </Accordion>
</AccordionGroup>

## Safety Mechanisms

<Note>
The app uses "safe defaults" when detection is uncertain:

```python
try:
    out = nc.send_command("show ip route connected", delay_factor=2)
    return bool(re.search(r'^C\s+', out, re.MULTILINE))
except Exception:
    return False   # If it fails, assume L2 for safety
```

**Rationale:** It's safer to omit L3 features (which can be added later) than to
send invalid commands that could disrupt switch operation.
</Note>
